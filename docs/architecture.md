# Описание приложения

## Что такое Locstep

Приложение для построения маршрутов по зданию на основе графовой модели. Пользователь задаёт точку «откуда» и «куда»; приложение находит кратчайший путь алгоритмом BFS и выдаёт пошаговые инструкции. Есть редактор графов (узлы и связи), поиск по названиям и алиасам, несколько карт, импорт/экспорт в JSON, локализация (RU/EN) и настройки темы.

## Модель данных и хранилище

Модель данных состоит из двух основных классов: **Node** и **BuildingGraph**.

**Node** — узел графа, immutable-объект. Поля: `id` (уникальный идентификатор), `name` (название для отображения), `aliases` (список алиасов для поиска), `neighbors` (список ID соседних узлов), `imageUrls` и `tour3dUrls` (опциональные ссылки на медиа).

**BuildingGraph** инкапсулирует граф. Узлы хранятся в `Map<String, Node>` — доступ по ID за O(1). Основные методы: `addNode()` — добавление узла, `removeNode()` — удаление с каскадным удалением ссылок из соседей, `addConnection()` — двунаправленная связь между двумя узлами, `findNode()` — поиск по запросу с учётом алиасов.

Поиск узлов реализован в **matches()** у Node: поддерживается запрос из нескольких слов; каждый токен должен встречаться в названии, ID или алиасах, порядок слов не важен. Пример: запрос «2 Юж» находит узел «Южная лестница 2».

Хранилище: **GraphStorage** (статические методы) сохраняет и загружает **GraphCollection** через SharedPreferences; коллекция хранит несколько графов и активный граф. Импорт/экспорт — чтение/запись JSON-файлов (file_picker).

## Алгоритм поиска пути

Используется **BFS (Breadth-First Search)**. Причины: граф невзвешенный (все переходы = 1 шаг), эвристика и веса не нужны, BFS гарантирует кратчайший путь по количеству шагов, сложность O(V + E); для типичного графа здания (до сотен/тысяч узлов) этого достаточно.

Алгоритм: инициализируется очередь с начальным узлом и словарь посещённых (nodeId → previousNodeId). На каждой итерации из очереди извлекается текущий узел. Если он целевой — путь восстанавливается по словарю и возвращается PathResult; иначе все непосещённые соседи помечаются, записываются в словарь и добавляются в очередь. Если очередь опустела — путь не найден.

**InstructionBuilder** превращает PathResult в пошаговые инструкции: для каждого шага маршрута — порядковый номер и название узла. Тексты локализованы (русский/английский).

## Слои архитектуры

- **UI:** NavigationScreen (построение маршрута), GraphEditorScreen (редактор графов), SettingsScreen (настройки).
- **Providers:** ThemeProvider (тема, светлая/тёмная/системная, цветовая схема), LocaleProvider (язык), TextScaleProvider (масштаб текста), Confirm3dTourProvider (спрашивать ли перед открытием ссылки на 3D-тур).
- **Engine:** PathFinder (BFS), InstructionBuilder (инструкции из пути), AliasResolver (автогенерация алиасов по названию).
- **Data:** GraphStorage (SharedPreferences, импорт/экспорт файлы), ImageCacheService (кэш изображений узлов).
- **Models:** BuildingGraph, Node, GraphCollection.

Поток: UI использует провайдеры и движок; движок читает модели и данные; данные сохраняют/загружают модели.

## Пользовательский интерфейс

Реализован на Flutter Widgets.

**NavigationScreen** — главный экран: CustomScrollView с SliverAppBar (растягивающаяся шапка), форма построения маршрута в Card с полями NodeAutocompleteField (автодополнение по узлам графа), отображение инструкций по маршруту.

**GraphEditorScreen** — редактор графа: TabController (вкладки «Узлы», «Инфо»). На вкладке «Узлы» — список узлов с раскрытием, просмотр соседей и управление связями. На больших экранах (ширина > 840 dp) — адаптивный layout с NavigationRail вместо нижнего TabBar.

**SettingsScreen** — секция «Внешний вид»: язык, тема (светлая/тёмная/системная), палитра, масштаб текста; секция «Поведение»: подтверждение перед открытием 3D-тура; секция «О приложении»: информация о приложении.

Состояние приложения управляется через **Provider:** ThemeProvider, LocaleProvider, TextScaleProvider — каждый хранит свои настройки и уведомляет UI об изменениях.

## Алиасы и поиск

**AliasResolver** генерирует алиасы по названию узла по предопределённым правилам (аудитория ↔ room, лестница ↔ stairs, направления северная ↔ north и т.д.). Пользователь может добавлять свои алиасы в редакторе. findNode() ищет по ID (точное), по name, по алиасам и по совпадению токенов в name/id/aliases (matches).

## Технологический стек

Flutter (Dart), Provider (состояние), SharedPreferences (хранение графов), file_picker (импорт/экспорт), flutter_localizations и intl (локализация RU/EN), google_fonts (Nunito), package_info_plus (версия в настройках), url_launcher (внешние ссылки). Material 3, адаптивная вёрстка под телефон и планшет/десктоп.

## Формат данных

Граф экспортируется в JSON: объект с полями `graph` (id, name, createdAt, modifiedAt, nodes) и `exportedAt`. Каждый узел в `nodes`: id, name, aliases, neighbors, при наличии imageUrls и tour3dUrls. Импорт читает такой же формат из файла.
